# Story 10.1 — Schema Supabase Otimizado para Acesso de Agentes

## Status: Pending

## Contexto
Para que os agentes AIOS possam criar e atualizar tasks via Supabase MCP, o schema precisa de políticas RLS específicas que permitam acesso autenticado por service_role, além de funções helper para operações comuns dos agentes.

## Acceptance Criteria

- [ ] AC1: Tabelas `company_tasks` e `task_comments` acessíveis via service_role (para agentes)
- [ ] AC2: Função Supabase `create_agent_task(board_id, title, assignee_id, priority, description)` disponível
- [ ] AC3: Função Supabase `update_task_status(task_id, new_status, agent_comment)` disponível
- [ ] AC4: Tabela `task_activity_log` criada para rastrear ações dos agentes
- [ ] AC5: Cada ação de agente fica registrada com: agent_id, action_type, task_id, timestamp, payload
- [ ] AC6: View `agent_dashboard` com tasks por assignee e status atual

## Schema Novo

```sql
-- Comentários de tasks
CREATE TABLE task_comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID REFERENCES company_tasks(id) ON DELETE CASCADE,
  author_id TEXT NOT NULL, -- agent ID ou user ID
  author_name TEXT NOT NULL,
  author_avatar TEXT,
  content TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Log de atividade dos agentes
CREATE TABLE task_activity_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID REFERENCES company_tasks(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL,
  agent_name TEXT NOT NULL,
  action_type TEXT NOT NULL, -- 'created' | 'status_changed' | 'commented' | 'assigned'
  old_value TEXT,
  new_value TEXT,
  payload JSONB,
  created_at TIMESTAMPTZ DEFAULT now()
);
```
